# Writing APIs for Native Client Schema Generation

When you build your Blocks app, the build pipeline generates `aws-blocks/blocks.spec.json` — an OpenRPC document describing every method in your `ApiNamespace`. Native client codegen tools (Swift, Kotlin, Dart) read this document to produce typed SDKs.

The spec is generated by a dedicated CLI, not by `npm run dev`. The CLI ships as a `bin` entry on `@aws-blocks/core`, so once your Blocks app has its dependencies installed it's available via `npx`:

```bash
npx blocks-generate-spec
```

By default it reads `./aws-blocks/index.ts` and writes `./aws-blocks/blocks.spec.json`. Pass alternate paths positionally if you need to:

```bash
npx blocks-generate-spec ./aws-blocks/index.ts ./aws-blocks/blocks.spec.json
```

To run it on every build, add it to your app's `package.json`:

```json
{
  "scripts": {
    "build:spec": "blocks-generate-spec",
    "build": "npm run build:spec && tsc && vite build"
  }
}
```

Why the dedicated CLI? The emitter loads the TypeScript compiler to introspect method signatures (~1.5s cold start). Wiring that into the dev server loop slows every restart and races short-lived test harnesses, so it's run as a separate build step instead.

The quality of the generated native client depends on the quality of the schema. This guide covers how to write your `aws-blocks/index.ts` so the spec emitter produces the best possible output.

---

## How the spec emitter resolves types

The emitter uses three tiers, in order of precision:

1. **Zod schemas** — if a method calls `.parse()` on an exported Zod schema, the emitter extracts full JSON Schema with constraints (`format: uuid`, `maxLength`, `minLength`, etc.)
2. **TypeScript types** — if no Zod schema is found, the emitter uses the TypeScript type checker to extract types from method signatures and inferred return values
3. **`unknown`** — if both fail, the param or return type is emitted as `{ "type": "unknown" }`, which generates `Any` in native clients

Most methods get good types from tier 2 automatically. Tier 1 is for when you need precision beyond what TypeScript can express.

## The basics: TypeScript types work automatically

Plain TypeScript method signatures produce correct base types with no extra work:

```ts
export const api = new ApiNamespace(scope, 'api', (context) => ({
  // ✅ Params: key → string. Return: string | null.
  async getUser(id: string) {
    return await store.get(id);
  },

  // ✅ Params: id → string, name → string, age → number.
  //    Return: { success: boolean }.
  async createUser(id: string, name: string, age: number) {
    await store.put(id, JSON.stringify({ name, age }));
    return { success: true };
  },

  // ✅ Return type inferred as { id: string; name: string; age: number }[]
  async listUsers() {
    const users: { id: string; name: string; age: number }[] = [];
    for await (const entry of store.scan()) {
      users.push({ key: entry.key, ...JSON.parse(entry.value) });
    }
    return users;
  },
}));
```

The emitter uses `ts.createProgram` with a full type checker, so it resolves imported types, type aliases, and inferred return types — you don't need explicit return type annotations.

## Best practices for TypeScript-only methods

### Use specific types, not `any`

```ts
// ❌ Bad — param and return both emit as unknown/any
async updateUser(id: any, data: any) {
  return await store.put(id, data);
}

// ✅ Good — precise types in the spec
async updateUser(id: string, data: { name: string; age: number }) {
  return await store.put(id, JSON.stringify(data));
}
```

### Prefer inline object types over unresolvable generics

```ts
// ⚠️ Complex generic — may degrade to { type: "object" }
async getItem(...args: Parameters<typeof table.get>) {
  return await table.get(...args);
}

// ✅ Explicit params — clean schema output
async getItem(pk: string, sk: string) {
  return await table.get({ pk, sk });
}
```

### Use `null` for absent values, not `undefined`

JSON has `null` but not `undefined`. The spec emitter maps both to `oneOf` with `{ type: "null" }`, but `null` is more explicit and idiomatic:

```ts
// ✅ Clear nullable in the spec
async getUser(id: string): Promise<User | null> {
  return await store.get(id) ?? null;
}
```

### Define shared types as named interfaces

Named types get extracted to `components.schemas` with `$ref` pointers, so native codegen produces named structs/classes instead of anonymous ones. If the same type appears in multiple methods, it's defined once and referenced everywhere.

Inline object literals (e.g., returning `{ userId, title, completed }` directly) produce anonymous types — the spec emitter inlines the schema on every method that returns that shape. Native codegen creates duplicate anonymous structs like `CreateTodoResult` and `ListTodosResultItem` with identical fields. Use a named interface and annotate your return types:

```ts
// ❌ Inline literal — duplicated anonymous schema per method
async createTodo(title: string) {
  return { userId: 'alice', todoId: '123', title, completed: false };
}
async listTodos() {
  return [{ userId: 'alice', todoId: '123', title: 'Buy milk', completed: false }];
}

// ✅ Named interface — one schema in components.schemas, $ref everywhere
interface Todo {
  userId: string;
  todoId: string;
  title: string;
  completed: boolean;
}

async createTodo(title: string): Promise<Todo> { ... }
async listTodos(): Promise<Todo[]> { ... }
```

### Use discriminated unions for action-style params

When a method accepts an `action` string and a bag of fields, native codegen produces a loose `(String, Map<String, String>)` signature. Use a discriminated union instead — each variant carries exactly the fields it needs:

```ts
// ❌ Loose — native clients get (String, Map<String, String>)
async setAuthState(action: string, fields: Record<string, string>) { ... }

// ✅ Discriminated union — native clients get per-action structs
type AuthActionInput =
  | { action: 'signIn'; username: string; password: string }
  | { action: 'signUp'; username: string; password: string }
  | { action: 'signOut' };

async setAuthState(input: AuthActionInput) { ... }
```

The spec emitter produces a `oneOf` with each variant as a separate object schema, and `action` as a single-value enum discriminant. Native codegen can generate sealed classes / enums with per-variant fields.

#### Pattern: derive the union from a payload map

When you also want compile-time per-action narrowing for direct TS callers (without forcing them to spell out the discriminant inside their object literal in the type tests), keep the payload registry as a map and derive the union from it:

```ts
// One source of truth for "what fields each action needs"
export interface AuthActionPayloadMap {
  signIn: { username: string; password: string };
  signUp: { username: string; password: string } & Record<string, string>;
  signOut: {};
  // ...
}

// Discriminated union derived mechanically — no drift between map and union
export type AuthActionInput = {
  [K in keyof AuthActionPayloadMap]: { action: K } & AuthActionPayloadMap[K];
}[keyof AuthActionPayloadMap];

async setAuthState(input: AuthActionInput): Promise<AuthState> { ... }
```

Both shapes — the map and the derived union — are exported. Customer code that constructs a literal like `{ action: 'signIn', username, password }` gets full per-action narrowing because TS narrows on `input.action` inside the implementation's `switch`. The OpenRPC emitter sees the union and produces `oneOf` over per-action schemas. See `packages/auth-common/src/ui.ts` for the live pattern.

> **Note on empty payloads.** Use `{}` (empty object type) rather than `Record<string, never>` for actions with no extra fields. The intersection `{ action: K } & Record<string, never>` is uninhabited because the discriminator can't satisfy the `never` index signature.

### `Record<K, V>` emits `additionalProperties`

`Record<string, string>` and other index-signature types emit as `{ type: "object", additionalProperties: { type: "string" } }`. Native codegen produces `Map<String, String>` / `Dictionary<String, String>`. This is correct but less precise than named fields — prefer explicit object types when the keys are known:

```ts
// ⚠️ Emits as Map<String, String> — keys are untyped
async update(fields: Record<string, string>) { ... }

// ✅ Emits as a named struct with known fields
async update(fields: { name: string; email: string }) { ... }
```

### Building Block helper methods work automatically

Methods created by Building Block helpers like `auth.createApi()` are resolved via the TypeScript type checker. As long as the helper method has proper return types, the emitter extracts full param and return type info:

```ts
// ✅ Types resolved automatically — no extra work needed
export const authApi = auth.createApi();
```

If the helper returns `any` or an unresolvable generic, the methods will emit as `unknown`. In that case, wrap them in an explicit `ApiNamespace`.

### Returning Building Block handles (Realtime, FileBucket)

Some BBs return live handles instead of plain data — `RealtimeChannel`, `FileDownloadHandle`, `FileUploadHandle`. These implement `toJSON()` returning a `{ __blocks: '<tag>' }` descriptor. The spec emitter detects the marker and lowers it onto the result schema as `x-blocks-transferable`, plus `x-blocks-type-args` for any generic arguments. Native codegen reads those fields and emits the matching SDK type (`RealtimeChannel<Cursor>` instead of a raw struct).

```ts
// ✅ Just return the handle — codegen handles the rest
async realtimeGetCursorChannel() {
	return realtime.getChannel<Cursor>('cursors', 'default');
}

async fileGetHandle(path: string) {
	return await bucket.getFileHandle(path);
}

async fileCreateUploadHandle(path: string, contentType?: string) {
	return await bucket.createUploadHandle(path, contentType ? { contentType } : undefined);
}
```

You don't opt in or annotate anything — if you return a BB handle, the spec carries the transferable tag automatically. See the [spec extensions reference](#spec-extensions-reference) below for the full list of tags and how native codegen consumes them.

## Adding Zod for precision (when you need it)

TypeScript types give you base types (`string`, `number`, `boolean`). When native clients need more — format constraints, length limits, patterns — add Zod schemas.

### Step 1: Define and export the schema

The schema must be **exported** from the module so the spec emitter can find it:

```ts
import { z } from 'zod';

// ✅ Exported — the spec emitter can detect this
export const CreatePostInput = z.object({
  title: z.string().min(1).max(120),
  body: z.string(),
  tags: z.array(z.string()).max(10).optional(),
}).meta({ id: 'CreatePostInput' });
```

### Step 2: Use `.parse()` in the method body

The spec emitter scans method source text for `.parse()` calls referencing exported schemas:

```ts
export const api = new ApiNamespace(scope, 'api', (context) => ({
  async createPost(rawInput: unknown) {
    const { title, body, tags } = CreatePostInput.parse(rawInput);
    // ...
  },
}));
```

This produces:

```json
{
  "name": "api.createPost",
  "params": [
    { "name": "title", "required": true, "schema": { "type": "string", "minLength": 1, "maxLength": 120 } },
    { "name": "body", "required": true, "schema": { "type": "string" } },
    { "name": "tags", "required": false, "schema": { "type": "array", "items": { "type": "string" }, "maxItems": 10 } }
  ]
}
```

### Step 3 (optional): Name the schema with `.meta()`

`.meta({ id: 'CreatePostInput' })` places the schema in `components.schemas` with that name. Without it, the schema properties are inlined directly on the params. Named schemas are reusable across methods and produce named types in native clients.

### When to use Zod vs. plain TypeScript

| Scenario | Recommendation |
|---|---|
| Internal API, TS frontend only | Plain TypeScript is fine |
| Native clients need basic types | Plain TypeScript works |
| Need `format: uuid`, `maxLength`, `pattern` | Add Zod |
| Need runtime input validation | Add Zod (you get both validation and schema) |
| Need named reusable types in native clients | Add Zod with `.meta({ id })` |

## Patterns that don't work well

### Spread/rest params

```ts
// ⚠️ Rest params emit a single "args" param with unknown type
async tablePut(...args: Parameters<typeof table.put>) {
  await table.put(...args);
}
```

The spec emitter sees one param named `args`. Prefer explicit params.

### `as any` type casts

```ts
// ❌ Casts erase type information
async putData(key: string, value: unknown) {
  await store.put(key, value as any);
  return { success: true };
}
```

The `value` param emits as `unknown`. Use a Zod schema or a specific type instead.

### `JSON.parse()` returns `any`

```ts
// ⚠️ Return type is `any` — emits as empty schema {} (means "any value")
async getResult(key: string) {
  const raw = await store.get(key);
  return raw ? JSON.parse(raw) : null;
}

// ✅ Add a type assertion or explicit return type
async getResult(key: string): Promise<JobResult | null> {
  const raw = await store.get(key);
  return raw ? JSON.parse(raw) as JobResult : null;
}
```

`JSON.parse()` returns `any`, which the spec emitter represents as `{}` (empty schema — "any value is valid"). Native codegen produces an untyped result. Add a return type annotation or use Zod to parse the JSON.

## Checking your spec output

After running `npx blocks-generate-spec`, inspect `aws-blocks/blocks.spec.json`:

```bash
# Count methods
node -e "const d=require('./aws-blocks/blocks.spec.json'); console.log(d.methods.length, 'methods')"

# Find methods with unknown params
node -e "const d=require('./aws-blocks/blocks.spec.json'); \
  d.methods.forEach(m => m.params.forEach(p => { \
    if(p.schema.type==='unknown') console.log(m.name+'.'+p.name) \
  }))"

# Find methods with unknown return types
node -e "const d=require('./aws-blocks/blocks.spec.json'); \
  d.methods.filter(m=>m.result.schema.type==='unknown').forEach(m=>console.log(m.name))"
```

If a method shows up with `unknown` types and you didn't expect it, check whether:
- The param is typed as `any` or `unknown` in the source
- The method uses rest/spread params
- The return type comes from `JSON.parse()` without a type annotation
- The helper method that creates the `ApiNamespace` returns `any`

## Spec extensions reference

This section is for native codegen authors and platform SDK authors. App developers don't need to read it — the extensions are emitted automatically based on what your `aws-blocks/index.ts` returns.

Blocks uses two OpenRPC `x-` extension fields. All others in `blocks.spec.json` are standard OpenRPC 1.3.2.

### `x-blocks-transferable` (on a method's `result.schema`)

Marks a result as a live handle that the platform SDK must hydrate at runtime, not a plain data class. The value is the descriptor tag — the same string the runtime puts in the descriptor's `__blocks` field.

```json
{
	"name": "api.realtimeGetCursorChannel",
	"result": {
		"name": "RealtimeGetCursorChannelResult",
		"schema": {
			"x-blocks-transferable": "realtime/channel",
			"x-blocks-type-args": [
				{ "$ref": "#/components/schemas/Cursor" }
			]
		}
	}
}
```

Tags currently emitted:

| Tag | Native SDK type to emit | Source BB |
|---|---|---|
| `realtime/channel` | `RealtimeChannel<T>` (where `T` comes from `x-blocks-type-args[0]`) | `bb-realtime` |
| `file-bucket/download` | `FileDownloadHandle` | `bb-file-bucket` |
| `file-bucket/upload` | `FileUploadHandle` | `bb-file-bucket` |

**Codegen contract.** When `x-blocks-transferable` is present on a result schema, ignore any other properties on the schema and emit the SDK type for that tag. The platform SDK is responsible for hydrating the descriptor returned over the wire into a live handle.

**Stability.** Tags are stable across versions. Adding a new tag is non-breaking; removing or renaming an existing tag is breaking and requires coordination with native SDK teams.

### `x-blocks-type-args` (on a method's `result.schema`)

Carries the JSON Schema for each generic type argument of the SDK type, in declaration order. Only meaningful alongside `x-blocks-transferable`.

```json
{
	"x-blocks-transferable": "realtime/channel",
	"x-blocks-type-args": [
		{ "$ref": "#/components/schemas/Cursor" }
	]
}
```

Each entry is a valid JSON Schema and may use `$ref` against `components.schemas`. For non-generic transferables (`file-bucket/download`, `file-bucket/upload` today), this field is omitted.

### What's not in the spec

Two things native SDK authors need that the spec deliberately does not carry:

1. **Wire-level protocol details** (WebSocket subprotocol names, message frames, auth flow) for non-RPC transports like Realtime. These live in per-BB protocol docs (e.g. `packages/bb-realtime/PROTOCOL.md`). Embedding them in the spec as static JSON would be docs-with-extra-steps — same content, harder to read.
2. **Per-environment endpoint URLs.** The spec's `servers` array may contain a snapshot from the local sandbox config, but it's not a multi-environment registry. Production, staging, and local URLs are managed by the platform SDK's runtime config.
